Neural Processing Unit

發表於 2025-12-22 更新於 2026-01-07 分類於 Computer Science ， Artificial Intelligence 文章字數： 3.8k 所需閱讀時間 ≈ 7 分鐘

NPU (Neural Processing Unit)，中文稱為「神經網路處理單元」，是一種專門為了加速人工智慧 (AI) 和機器學習 (Machine Learning) 運算而設計的微處理器。

如果說 CPU 是通才，GPU 是繪圖專家，那麼 NPU 就是專門為 AI 數學運算打造的數學天才。

為什麼我們需要 NPU？

在 NPU 出現之前，AI 運算主要依賴 CPU 和 GPU，但它們面臨瓶頸：

CPU (中央處理器)： 擅長序列處理和邏輯控制，但核心數少，處理大量並行運算（矩陣乘法）效率低下。
GPU (圖形處理器)： 雖然擅長並行運算，但功耗較高，且架構主要為圖形渲染設計，對於某些 AI 推論 (Inference) 任務來說，能效比（Performance per Watt）不是最佳。

NPU 的優勢在於：

  1. 高效率： 專門針對神經網路的「乘積累加運算 (MAC)」進行硬體優化。
  2. 低功耗： 在處理相同 AI 任務時，NPU 的耗電量遠低於 GPU，這對手機、筆電等電池供電設備至關重要。
  3. 專用架構： 內建專用的快取 (Cache) 和資料流架構，減少資料搬運的延遲。

NPU 在 AI PC 和 Physical AI 中的角色

隨著 AI 應用的普及，NPU 的地位急劇上升，特別是在以下領域：

A. AI PC (AI 電腦)

微軟定義的 AI PC 要求必須配備算力達到 40 TOPS (Trillion Operations Per Second) 以上的 NPU。

  • 用途： 在不連網的情況下，本機執行 Microsoft Copilot、自動去背、即時視訊模糊、語音降噪。
  • 代表產品： Intel Core Ultra (Meteor Lake/Lunar Lake), AMD Ryzen AI, Qualcomm Snapdragon X Elite, Apple M 系列晶片 (Neural Engine)。

B. Physical AI (實體 AI)

在機器人與邊緣裝置中，NPU 是大腦的關鍵部分。

  • 即時性： 機器人看到障礙物必須在毫秒級內反應，NPU 的低延遲特性至關重要。
  • 離線運行： 自駕車或工廠機器人不能依賴不穩定的雲端網路，必須靠本地端的 NPU 進行推論。

C. 智慧型手機

  • 用途： 臉部解鎖 (FaceID)、夜景模式攝影運算、即時翻譯、語音助理喚醒。

常見的 NPU 術語與指標

  • TOPS (Trillion Operations Per Second)： 這是衡量 NPU 效能的最常見單位。代表每秒能進行多少兆次運算。
    • 例如：Snapdragon X Elite 的 NPU 算力約為 45 TOPS。
  • INT8 / FP16： NPU 支援的資料精度。
    • INT8 (8-bit 整數)： 精度較低但速度極快，省電，適合手機端的 AI 推論。
    • FP16 (16-bit 浮點數)： 精度較高，適合需要準確度的運算。
  • 異質運算 (Heterogeneous Computing)： 現代系統會協調 CPU、GPU 和 NPU。
    • CPU 負責啟動程式 GPU 負責大型訓練或高負載渲染 NPU 負責持續性的背景 AI 任務（如視訊會議眼神接觸修正）。

總結：處理器三足鼎立的時代

計算機架構已經從過去的 CPU 獨大，演變為 CPU + GPU + NPU 的黃金三角：

處理器 專長 AI 角色
CPU 邏輯、控制、序列任務 總指揮，處理小量或複雜邏輯的 AI
GPU 大規模並行、浮點運算 AI 模型「訓練 (Training)」的主力
NPU 矩陣運算、低功耗 AI 模型「推論 (Inference)」的最佳選擇

隨著 Physical AI 和生成式 AI 進入終端裝置，NPU 將像當年的 GPU 一樣，成為所有電子產品的標準配備。

General Process 通用流程（不管哪家 NPU 都差不多）：

  1. 先選模型與框架（如 PyTorch 或 TensorFlow），將模型匯出成部署格式（常見為 ONNX 或 TFLite），
  2. 接著進行 NPU 友善化處理——優先做 INT8 量化（多數 NPU 推論以低精度為主），並避免或替換 NPU 不支援的算子以免被 fallback 到 CPU/GPU；
  3. 然後使用對應的 runtime 啟用 NPU 後端（ONNX Runtime 使用 Execution Provider、TFLite 使用 Delegate，或依廠商提供的 SDK 如 RKNN、Hailo、EdgeTPU 等進行編譯與部署），
  4. 最後驗證是否真的在 NPU 上運行（檢查 runtime log 看哪些節點/算子被 offload，並比對 CPU 與 NPU 的 latency）。

    在 iPhone 15（使用 Apple Neural Engine, ANE）上做 NPU 推論的通用流程是：先在開發機把模型從原始框架（PyTorch/TF/ONNX）用 coremltools 轉成 Core ML 格式（.mlmodel），並在轉換時做必要的優化與量化（常見為 float16 或 8‑bit 靜態量化並用代表性資料做 calibration），注意避免或以 Core ML 支援的方式改寫不支援的自訂算子，之後由 Xcode 編譯成 .mlmodelc 以減少執行時開銷，將編譯後模型放入 app 資源並在程式中以 MLModelConfiguration 設定 computeUnits = .all（或 .cpuAndGPU / .cpuOnly 作比較）以啟用 ANE，使用自動生成的 model class 或 MLModel API 進行同步／非同步推論與批次處理；驗證與剖析時用 Instruments（Time Profiler / Core ML instrument）、Xcode Console 與 A/B 測試（切換 computeUnits）來比對 CPU/GPU/ANE latency 與耗能，並檢查轉換或執行過程的相容性警告以確保沒有算子回落到 CPU，若需更高效能可進一步在轉換階段做 operator fusion、調整資料 layout（如 NHWC）、預熱模型（warm‑up）、以及重用記憶體與 batch 策略；若你提供模型格式與類型（影像分類/物件檢測/語音/文本）我可以產出具體 coremltools 轉換命令、量化範例與可直接貼入 Xcode 的 Swift 範例程式碼。
