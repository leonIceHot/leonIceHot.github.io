Neural Processing Unit

發表於 2025-12-22 更新於 2026-01-07 分類於 Computer Science ， Artificial Intelligence 文章字數： 3.5k 所需閱讀時間 ≈ 6 分鐘

Neural Processing Unit（NPU，神經網路處理器）是為深度學習推論（與部分訓練）量身設計的專用加速器，透過大量乘加單元（MAC/PE 陣列）、片上 SRAM、有效的 NoC/互連與 DMA、以及低精度算術（如 INT8、FP16、BF16 等）來大幅提升每瓦效能；其設計核心在於以 dataflow（例如 weight‑stationary、output‑stationary、row‑stationary）與空間/時間切片最大化資料重用、減少與 DRAM 的資料搬移，並透過 compiler/graph optimizer（operator fusion、layout transform、量化）與 runtime（execution provider / delegate）把模型分割成 NPU 可支援的子圖以下放算子；實務上常見工作流程包含模型匯出（PyTorch/TF → ONNX/TFLite）、量化（PTQ 或 QAT）、用 vendor SDK/工具將模型編譯成 NPU 格式並在 target device 上以 runtime 執行與剖析（確認 offload 與效能），衡量指標有 latency、throughput（GOPS）、energy per inference（J/inference 或 TOPS/W）、memory bandwidth utilization 與量化後的 accuracy drop；主要挑戰為記憶體牆、模型多樣性與稀疏性支援、可程式化與專用化間的取捨，以及系統整合（CPU/GPU 協作、QoS、安全性）；近期趨勢包括針對 Transformer/LLM 的優化、FP8 與更低精度支援、near‑memory / in‑memory compute、以及更強的 compiler 自動化與稀疏友好硬體支援，代表性架構有 DianNao/DaDianNao、Eyeriss、Google TPU、EIE、SCNN 與 MAERI 等，可依目標應用（edge 或 cloud）與硬體平台選擇相應的 SDK 與最佳化流程以達到最佳推論效能。

General Process 通用流程（不管哪家 NPU 都差不多）：

  1. 先選模型與框架（如 PyTorch 或 TensorFlow），將模型匯出成部署格式（常見為 ONNX 或 TFLite），
  2. 接著進行 NPU 友善化處理——優先做 INT8 量化（多數 NPU 推論以低精度為主），並避免或替換 NPU 不支援的算子以免被 fallback 到 CPU/GPU；
  3. 然後使用對應的 runtime 啟用 NPU 後端（ONNX Runtime 使用 Execution Provider、TFLite 使用 Delegate，或依廠商提供的 SDK 如 RKNN、Hailo、EdgeTPU 等進行編譯與部署），
  4. 最後驗證是否真的在 NPU 上運行（檢查 runtime log 看哪些節點/算子被 offload，並比對 CPU 與 NPU 的 latency）。

Mobile

Android

  1. Galaxy S25 Ultra
  2. Open Development Mode
    https://www.samsung.com/uk/support/mobile-devices/how-do-i-turn-on-the-developer-options-menu-on-my-samsung-galaxy-device/
  3. Process
    在 Galaxy S25 Ultra 上做 NPU 開發的通用實務流程是：先用設定頁或 adb（例如 adb shell getprop ro.board.platform 等）確認手機 SoC 與系統版本，接著準備 Android Studio（含 SDK/NDK）與 adb、開啟手機的開發者選項與 USB 偵錯，然後選擇推論堆疊—最容易跨平台的是走 Android NNAPI（通常透過 TensorFlow Lite 的 NNAPI Delegate 讓系統自動把可支援算子下放到 NPU），若需要更極致效能或特殊算子支援再改用 SoC/廠商專屬 SDK（例如對應平台的工具鏈與 runtime）；模型端先在 PyTorch/TF 訓練或取得模型後匯出成部署格式（常見 ONNX 或直接轉 TFLite），再做 NPU 友善化與最佳化（優先 INT8 量化並用代表性資料校正、盡量使用 NPU 支援的算子與資料格式、避免不支援算子以免 fallback 到 CPU/GPU、必要時做 op fusion/graph 優化與輸入尺寸調整），在 App 端用對應 runtime 啟用硬體後端（TFLite 加 NNAPI Delegate，或 ONNX Runtime 使用 Execution Provider，或改走廠商 runtime），最後用 runtime log/除錯訊息確認哪些節點真的被 offload 到 NPU，並以 A/B 測試比較 CPU vs NPU 的 latency、throughput 與功耗（可搭配 Android Studio Profiler/Perfetto）來定位瓶頸與迭代優化。

IOS

  1. iPhone 15
  2. Process
    在 iPhone 15（使用 Apple Neural Engine, ANE）上做 NPU 推論的通用流程是：先在開發機把模型從原始框架（PyTorch/TF/ONNX）用 coremltools 轉成 Core ML 格式（.mlmodel），並在轉換時做必要的優化與量化（常見為 float16 或 8‑bit 靜態量化並用代表性資料做 calibration），注意避免或以 Core ML 支援的方式改寫不支援的自訂算子，之後由 Xcode 編譯成 .mlmodelc 以減少執行時開銷，將編譯後模型放入 app 資源並在程式中以 MLModelConfiguration 設定 computeUnits = .all（或 .cpuAndGPU / .cpuOnly 作比較）以啟用 ANE，使用自動生成的 model class 或 MLModel API 進行同步／非同步推論與批次處理；驗證與剖析時用 Instruments（Time Profiler / Core ML instrument）、Xcode Console 與 A/B 測試（切換 computeUnits）來比對 CPU/GPU/ANE latency 與耗能，並檢查轉換或執行過程的相容性警告以確保沒有算子回落到 CPU，若需更高效能可進一步在轉換階段做 operator fusion、調整資料 layout（如 NHWC）、預熱模型（warm‑up）、以及重用記憶體與 batch 策略；若你提供模型格式與類型（影像分類/物件檢測/語音/文本）我可以產出具體 coremltools 轉換命令、量化範例與可直接貼入 Xcode 的 Swift 範例程式碼。
