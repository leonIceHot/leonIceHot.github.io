Architecture Style

發表於 2025-01-19 更新於 2025-01-26 分類於 Software Engineering ， Software Architecture 文章字數： 2.6k 所需閱讀時間 ≈ 5 分鐘

軟體架構風格（Software Architecture Style）是指在設計軟體系統時，為解決特定類型的問題而採用的一組設計模式和規範的集合。它描述了系統內各部分的組織方式以及它們之間的互動關係。

軟體架構風格介紹

1. 階層式架構（Layered Architecture）

  • 概念：將系統分為多個層級，每一層專注於特定功能，並且只與相鄰層進行互動。
  • 特徵：
    • 常見於 Web 開發中，如表示層（UI）、業務層（Business Logic）、資料層（Data Access）。
    • 易於維護，單一層的變更通常不影響其他層。
  • 應用：企業應用、網站。

2. 事件驅動架構（Event-Driven Architecture, EDA）

  • 概念：基於事件流的系統架構，模組間通過事件進行通訊。
  • 特徵：
    • 高度解耦，事件生產者與消費者無需直接相互依賴。
    • 支援非同步處理。
  • 應用：即時資訊系統、物聯網系統。

3. 微服務架構（Microservices Architecture）

  • 概念：將系統拆分為一組小型、獨立的服務，每個服務只負責單一功能。
  • 特徵：
    • 每個服務獨立部署，使用輕量級通訊協定（如 HTTP 或資訊隊列）。
    • 易於擴展和容錯。
  • 應用：大型分散式系統、電子商務平台。

4. 服務導向架構（Service-Oriented Architecture, SOA）

  • 概念：基於服務的設計風格，通過標準化的介面讓不同模組互動。
  • 特徵：
    • 使用 SOAP 或 REST 作為通訊協定。
    • 重視服務重用和集成。
  • 應用：企業集成解決方案、跨平台應用。

5. 管道-過濾器架構（Pipe-and-Filter Architecture）

  • 概念：資料以流的形式經過一系列過濾器進行處理，過濾器之間通過管道連接。
  • 特徵：
    • 每個過濾器專注於單一任務，結果通過管道傳遞給下一個過濾器。
    • 易於模組化和重用。
  • 應用：音訊或影像處理系統、資料流處理。

6. 事件溝通架構（Publish-Subscribe Architecture）

  • 概念：發布者（Publisher）向主題（Topic）發送資訊，訂閱者（Subscriber）根據主題接收相關資訊。
  • 特徵：
    • 支援多對多通訊。
    • 訂閱者和發布者彼此解耦。
  • 應用：新聞訂閱系統、分散式事件驅動架構。

7. 客戶端-伺服器架構（Client-Server Architecture）

  • 概念：將系統分為客戶端（Client）和伺服器（Server），客戶端負責向伺服器發送請求，伺服器負責處理並返回結果。
  • 特徵：
    • 客戶端與伺服器的功能分離。
    • 支援多個客戶端同時訪問。
  • 應用：Web 應用、移動應用。

8. REST 架構風格（Representational State Transfer）

  • 概念：基於 HTTP 協議的架構風格，遵循無狀態、資源導向的設計原則。
  • 特徵：
    • 使用標準 HTTP 方法（GET、POST、PUT、DELETE）。
    • 資源通過 URL 表示，資料以 JSON 或 XML 傳遞。
  • 應用：API 設計、微服務通訊。

如何選擇適合的軟體架構風格

1. 瞭解系統需求

  • 功能需求：系統需要實現什麼功能？（如：使用者界面、資料處理、通訊）
  • 非功能需求：考慮可擴展性、性能、安全性、可維護性和可靠性等。

2. 評估關鍵因素

  • 團隊專業知識：
    • 團隊是否對某些特定架構風格或工具有經驗？
  • 專案規模與複雜性：
    • 小型、簡單專案適合 單體架構（Monolithic） 或 階層式架構（Layered Architecture）。
    • 大型、分散式系統可能需要 微服務架構（Microservices Architecture） 或 事件驅動架構（Event-Driven Architecture）。
  • 可擴展性需求：
    • 水平擴展：選擇 微服務架構 或 無伺服器架構（Serverless Architecture）。
    • 垂直擴展：選擇 單體架構 或 客戶端-伺服器架構（Client-Server Architecture）。
  • 變更頻率：
    • 如果系統需求變更頻繁，選擇 微服務架構。
    • 如果系統相對穩定，選擇 階層式架構 或 單體架構。

3. 將風格匹配到問題

以下是常見架構風格與典型應用場景的對應關係：

架構風格 適用場景
階層式架構（Layered Architecture） 小型應用，功能清晰且需求穩定，例如簡單的企業應用或內部工具。
事件驅動架構（Event-Driven Architecture） 即時處理或非同步通訊的系統，例如即時資訊、物聯網應用。
微服務架構（Microservices Architecture） 大型模組化應用，每個模組獨立運行，例如電子商務平台或 SaaS 產品。
面向服務架構（SOA） 需要與遺留系統集成，或需要標準化服務介面的系統，例如企業應用集成方案。
客戶端-伺服器架構（Client-Server Architecture） 傳統兩層結構的系統，例如 Web 應用、移動應用後端。
REST 架構風格（REST Architecture Style） 需要輕量級、無狀態通訊的系統，例如開放 API 或微服務通訊。
管道-過濾器架構（Pipe-and-Filter Architecture） 資料流處理系統，例如音頻、視頻處理或大資料處理應用。
發布-訂閱架構（Publish-Subscribe Architecture） 訂閱者與發布者解耦的系統，例如新聞訂閱系統、分散式事件驅動系統。

4. 考慮取捨

  • 複雜性：
    • 簡單架構（如單體架構或階層式架構）易於開發和管理。
    • 分散式架構（如微服務架構）更適合高擴展性，但實現和維護更具挑戰性。
  • 性能：
    • 單體架構 在緊密耦合環境中通常性能更高。
    • 分散式系統 在處理多樣化工作負載或流量高峰時表現更優。
  • 成本：
    • 微服務架構 的建設和運維成本通常高於簡單架構。

5. 考慮未來需求

  • 系統是否會顯著增長或需要處理大量使用者？
  • 是否需要頻繁添加新功能或整合新技術？
  • 系統是否需要支持多租戶或全球使用者？

6. 迭代與最佳化

  • 可以從基線架構開始，根據反饋進行最佳化。
  • 在大型專案中，可以對子系統進行小規模原型設計，以評估可行性。

實例決策場景

  1. 小型電子商務網站：
    • 使用 階層式架構 管理使用者界面、業務邏輯和資料庫。
  2. 即時聊天應用：
    • 使用 事件驅動架構 支持非同步資訊處理。
  3. 大型 SaaS 平台：
    • 使用 微服務架構 以支持模組化和獨立部署。
  4. 資料處理系統：
    • 使用 管道-過濾器架構 進行大資料流處理。

通過系統化地分析需求，並結合架構風格的特點與應用場景，可以選擇最適合的架構風格來實現系統目標。
