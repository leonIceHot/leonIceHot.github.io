On-Device Inference

發表於 2026-04-09 更新於 2026-04-10 分類於 Computer Science ， Artificial Intelligence ， Edge AI 文章字數： 3.2k 所需閱讀時間 ≈ 6 分鐘

裝置端推理 (On-device Inference) 常被使用於兩大主流機器學習框架中：TensorFlow Lite (TFLite) 與 ONNX Runtime。

On-device Inference

以下是它在這兩個框架中如何運作的解析：

  1. TensorFlow Lite (GPU Delegate / GPU 委派)
    在 TFLite 中，「委派 (delegates)」機制是用來將模型（或模型的一部分）的執行任務轉交給硬體加速器來處理。

    • Android： GPU 委派依賴 OpenCL 或 OpenGL ES 來加速神經網路的執行。
    • iOS： GPU 委派依賴 Apple 的 Metal API。

  2. ONNX Runtime (Execution Providers / 執行提供者)
    ONNX Runtime 使用「Execution Provider (EP，執行提供者)」這個詞彙來代替「委派」，但兩者的核心概念是完全相同的。它會將運算操作指派（路由）給當下最有效率的硬體來執行。

  • NVIDIA GPU： 使用 CUDA 或 TensorRT 執行提供者。
    • Windows： 使用 DirectML 執行提供者（可支援並跨越 AMD、Intel 與 NVIDIA 的 GPU 運作）。
    • Apple： 使用 CoreML 執行提供者。

TensorFlow Lite

TensorFlow Lite (TFLite) 的 Delegate（委派）機制是一種硬體加速架構。
它的核心概念是：將神經網路模型中的某些計算工作，從通用的中央處理器（CPU）「委派」給專門的硬體加速器（例如 GPU、NPU、DSP 等）來執行，以達到更快的推理（Inference）速度和更低的耗電量。

預設情況下，TFLite 模型是完全在 CPU 上執行的。雖然 CPU 很通用，但對於神經網路中大量的矩陣乘法而言，效率不如專門的硬體。Delegate 機制就是為了解決這個問題而誕生的。

以下是 TFLite Delegate 機制的詳細運作原理與特點：

  1. 運作原理：圖分割（Graph Partitioning）
    當你載入一個 TFLite 模型並啟用 Delegate 時，TFLite 的直譯器（Interpreter）會進行以下步驟：

    • 檢查支援度： 直譯器會掃描模型中的所有操作節點（Operations / Ops），並詢問 Delegate：「你能處理這些操作嗎？」
    • 圖分割（Graph Partitioning）： 直譯器會將整個神經網路模型（Graph）切分成多個子圖（Subgraphs）。
    • Delegate 支援的操作會被歸類到 Delegate 子圖。
    • Delegate 不支援的操作會留在 CPU 子圖。
    • 執行時切換： 執行模型時，資料會先在 CPU 上處理，遇到 Delegate 子圖時，會將資料（Tensors）傳遞給加速器計算，計算完後如果後面還有 CPU 子圖，再將結果傳回 CPU 繼續運算。

  2. 常見的 TFLite Delegates
    TFLite 官方針對不同的平台提供了多種內建的 Delegate：

    • GPU Delegate： 將運算交給手機或電腦的顯示晶片。
    • Android 上底層使用 OpenCL 或 OpenGL ES，iOS 上底層使用 Metal API。
    • NNAPI Delegate (Android)： Android 平台的「神經網路 API」。它會自動尋找手機上最強的 AI 專屬晶片（如 NPU - 神經處理單元，或 TPU），如果沒有則退回 GPU 或 CPU。
    • Core ML Delegate (iOS)： 將運算交給 Apple 的 Core ML 框架，充分利用 iPhone/iPad 內的 Apple Neural Engine (ANE)。
    • Hexagon Delegate： 專門針對搭載高通（Qualcomm）Snapdragon 處理器的設備，將運算委派給 Hexagon DSP（數位訊號處理器），非常省電。

Hexagon Delegate

使用 Hexagon Delegate 是為了在搭載 Qualcomm（高通）Snapdragon 處理器的 Android 裝置上，利用內建的 Hexagon DSP（數位訊號處理器） 來加速 TensorFlow Lite 模型。

DSP 的最大優勢在於極低的功耗和針對整數運算的高效能，非常適合用於語音喚醒、持續的影像分類等需要長時間在背景運行的 AI 任務。

確保模型符合要求 (非常重要)

Hexagon DSP 的硬體架構是針對整數 (Integer) 運算最佳化的。因此，你的 TFLite 模型通常必須是經過量化 (Quantized) 的（例如 8-bit 整數 INT8 或 UINT8）。
如果你的模型是浮點數 (Float32) 模型，Hexagon Delegate 的加速效果會很差，甚至會直接退回 (Fallback) 給 CPU 執行。

注意事項與限制

  1. 硬體限制： 這只適用於高通 Snapdragon 處理器的裝置。如果在聯發科 (MediaTek)、三星 (Exynos) 或 Google Tensor 的晶片上執行，Hexagon Delegate 將無法運作，會發生錯誤或直接退回 CPU。
  2. 自動退回 (Fallback)： 如果模型中包含 Hexagon 不支援的算子 (Ops)，TFLite 會自動將該算子交回給 CPU 處理（即 Graph Partitioning）。
  3. 動態檢查： 實務開發上，建議先透過程式碼檢查裝置 SoC 的品牌，或者使用 Try-Catch 包裝。如果 Hexagon Delegate 初始化失敗，就優雅地退回使用 NNAPI Delegate 或 GPU Delegate。
